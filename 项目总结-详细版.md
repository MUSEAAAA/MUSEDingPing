## 1. 项目整体架构

### 这是什么项目？
一个**本地生活服务电商平台**的后端。用户可以：
- 浏览店铺（按分类、按名称搜索）
- 查看探店笔记（图文内容）
- 给笔记点赞
- 领取店铺优惠券
- **秒杀**限量优惠券（核心高并发场景）

```
Controller（接收HTTP请求）
    ↓
Service（业务逻辑层）
    ↓
Mapper（MyBatis-Plus，数据库访问）
    ↓
MySQL（持久化存储） + Redis（缓存/消息队列/分布式锁）
```

```yaml
server:
  port: 8081           # 后端端口

  redis:

    lettuce:           # Redis 连接池（Lettuce 是线程安全的 Redis 客户端）
```


## 2. 用户登录 — Session 与 Token 详解

### 传统 Session 登录（你代码里没用的方案）

```
用户登录 → 服务端创建 Session → 把 SessionID 写回 Cookie
下次请求 → 浏览器自动带 Cookie → 服务端根据 SessionID 查找 Session
```

**问题**：
1. Session 存在**服务端内存**里，用户量大了内存扛不住
2. 分布式部署时，用户请求可能打到不同服务器，Session 不同步（需要 Session 共享方案，如 Redis + Spring Session）

### 你用的 Token 登录（替代 Session）

```
用户登录 → 服务端生成随机 Token → 把用户信息存到 Redis（key=token, value=用户数据）
            → 把 Token 返回给前端
下次请求 → 前端把 Token 放到请求头 Authorization → 服务端根据 Token 查 Redis 拿到用户
```

**优点**：
- 服务端**不存 Session**（无状态 Token 存在 Redis）
- 多台服务器共享一个 Redis，天然解决分布式 Session 问题
- Token 过期时间由服务端控制（Redis TTL），可以随时续期

### 你代码里的具体实现

```java
// UserServiceImpl.java - login 方法

// 1. 生成随机 Token
String token = UUID.randomUUID().toString(true);

// 2. 将 User 转为 Map（Redis Hash 结构只能存字符串，所以所有字段 toString）
UserDTO userDTO = BeanUtil.copyProperties(user, UserDTO.class);
Map<String, Object> userMap = BeanUtil.beanToMap(userDTO, new HashMap<>(),
    CopyOptions.create()
        .setIgnoreNullValue(true)
        .setFieldValueEditor((fieldName, fieldValue) -> fieldValue.toString()));

// 3. 存入 Redis Hash
String tokenKey = "login:token:" + token;          // key
stringRedisTemplate.opsForHash().putAll(tokenKey, userMap);  // 用 Hash 存多个字段
stringRedisTemplate.expire(tokenKey, 36000, TimeUnit.MINUTES);  // 10 小时过期
```

### 验证码登录流程

```
1. 发送验证码 → 校验手机号格式 → 生成6位随机数 → 存 Redis（login:code:手机号, 2分钟过期）
2. 登录 → 用户输入手机号+验证码 → 从 Redis 取出验证码对比
        → 对的话查 MySQL 有没有这个用户
        → 没有就创建新用户（nickName = user_ + 随机字符串）
        → 生成 Token，用户信息写 Redis Hash，返回 Token
```

### 面试话术

> "我用 Token 替代了 Session 实现登录。用户登录后生成随机 UUID 作为 Token，把用户信息序列化成 Map 存到 Redis Hash 里，前端后续请求在 Header 里带 Token。这样做的好处是无状态、天然支持分布式部署，而且 Token 过期时间可以由服务端动态续期。"

### 复习方法
- 在浏览器按 F12 → Application → Cookies，看看你已经登录的网站是怎么存 SessionID 或 Token 的
- 画一张图：用户→前端→后端→Redis 的完整请求链路

## 3. 双拦截器机制 — 登录态校验与续期

### 为什么需要两个拦截器？

因为有一个**矛盾需求**：
- 大部分接口需要登录才能访问（`/blog/like` 等）
- 但是**登录接口本身**不需要登录（`/user/login`）
- 还有一个**中间层需求**：不管用户有没有登录，只要他带了 Token，我都应该帮他**刷新 Token 的过期时间**，防止用着用着突然过期

### 解决方案：两个拦截器 + 不同 order

```
第一个拦截器（order=0,拦截所有路径）: RefreshTokenInterceptor
    ↓ 只管刷新 Token，不管是否拦截
第二个拦截器（order=1,拦截除白名单外路径）: LoginInterceptor
    ↓ 检查有没有用户，没有就返回 401
```

### RefreshTokenInterceptor（刷新 Token）

```java
public boolean preHandle(...) {
    // 1. 取请求头里的 token
    String token = request.getHeader("authorization");
    if (token 为空) return true;  // 没有 Token？放行，让后面的拦截器决定

    // 2. 查 Redis
    Map<Object, Object> userMap = stringRedisTemplate.opsForHash().entries(key);

    // 3. 查到用户 → 写入 ThreadLocal
    UserHolder.saveUser(userDTO);

    // 4. 刷新 Token 过期时间（关键！）
    stringRedisTemplate.expire(key, LOGIN_USER_TTL, TimeUnit.MINUTES);
    return true;  // 不管有没有用户都放行
}
```

**关键设计**：这个拦截器**永不拦截**，只做两件事：查到用户就塞到 ThreadLocal、刷新 Token 过期时间。

### LoginInterceptor（校验登录）

```java
public boolean preHandle(...) {
    if (UserHolder.getUser() == null) {
        response.setStatus(401);  // 没有登录
        return false;
    }
    return true;
}
```

**关键设计**：只做一件事：查 ThreadLocal，没有用户就返回 401。因为 RefreshTokenInterceptor 已经把所有路径都处理过了，LoginInterceptor 只需要检查 ThreadLocal。

### 拦截器注册（MvcConfig.java）

```java
// 登录校验拦截器 — 排除登录/注册/首页等公开接口
registry.addInterceptor(new LoginInterceptor())
    .excludePathPatterns(
        "/shop/**", "/voucher/**", "/shop-type/**",
        "/upload/**", "/blog/hot",
        "/user/code", "/user/login"
    ).order(1);

// Token 刷新拦截器 — 拦截所有请求
registry.addInterceptor(new RefreshTokenInterceptor(stringRedisTemplate))
    .addPathPatterns("/**").order(0);
```

### 面试话术

> "我用了双拦截器设计。第一个拦截器负责从请求头解析 Token、恢复用户到 ThreadLocal、并刷新 Token 过期时间，它永远放行不拦截。第二个拦截器才做真正的登录校验，检查 ThreadLocal 有没有用户。这样做的好处是：用户只要在 Token 有效期内持续操作，过期时间会不断续期，不会突然掉登录。"

### 复习方法
- 关闭 Idea，在脑子里走一遍请求流程：请求进来 → order0 执行 → order1 执行 → Controller 执行
- 问自己：为什么不用一个拦截器解决所有问题？

## 4. ThreadLocal — 线程级数据隔离

### 为什么需要 ThreadLocal？

因为 HTTP 请求是在 Tomcat 的**线程池**里执行的，一个请求 = 一个线程。多个请求之间不能共享变量，否则数据会乱。但同一个请求的处理链条（拦截器 → Controller → Service）是在**同一个线程**里执行的，可以用 ThreadLocal 传递数据。

### UserHolder 的实现

```java
public class UserHolder {
    private static final ThreadLocal<UserDTO> tl = new ThreadLocal<>();

    public static void saveUser(UserDTO user) { tl.set(user); }
    public static UserDTO getUser() { return tl.get(); }
    public static void removeUser() { tl.remove(); }  // 防止内存泄漏！
}
```

### 关键点：为什么用 static

`ThreadLocal` 对象的 `static` 和 `非 static` 区别：

| 修饰符 | 效果 |
|--------|------|
| `static ThreadLocal` | 所有线程共享同一个 ThreadLocal 对象，但**每个线程有自己的副本** |
| 非 static ThreadLocal | 每个实例都有自己的 ThreadLocal |

通常**用 static**，因为 ThreadLocal 的语义就是"全局可见、线程隔离"。

### 为什么要在 afterCompletion 里 remove？

```java
@Override
public void afterCompletion(...) {
    UserHolder.removeUser();
}
```

Tomcat 使用**线程池**，线程处理完请求后不会销毁，而是归还到池中。如果不 remove，下一个使用这个线程的请求就能拿到上个请求的用户数据。这叫做**内存泄漏**（虽然是数据泄漏不是严格的内存泄漏，通常也叫这个名）。

### 面试话术

> "我使用 ThreadLocal 实现在一个请求周期内传递用户信息。拦截器解析 Token 后将用户对象存入 ThreadLocal，Controller 和 Service 层通过 UserHolder 直接获取，避免了在方法参数间层层传递。同时我会在请求完成后调用 remove() 清理，防止线程池复用导致的数据错乱。"

## 5. Redis 缓存 — 穿透/击穿/雪崩

### 5.1 缓存穿透

**问题**：请求一个**根本不存在的数据**（比如 id=-1 的店铺）。缓存里没有，每次请求都打到数据库。

**解决方案（空值缓存）**：

```java
// CacheClient.java - queryWithPassThrough

// 1. 查 Redis
String json = stringRedisTemplate.opsForValue().get(key);
if (json 不为空且不为空串) return 数据;     // 缓存命中

// 2. **关键：命中空串 — 说明之前已经查过DB，不存在**
if (json 等于 "") return null;              // 直接返回，不打DB

// 3. 查数据库
R r = dbFallback.apply(id);
if (r == null) {
    // 4. 缓存空值，防止下次继续打DB
    stringRedisTemplate.opsForValue().set(key, "", 2, TimeUnit.MINUTES);
    return null;
}
```

**原理**：把"数据不存在"这个结果也缓存起来，写入一个空字符串。下次同样的请求直接从 Redis 返回空，不会打到数据库。

### 5.2 缓存击穿

**问题**：一个**热点 key** 过期的一瞬间，大量并发请求同时打向数据库。

**解决方案（逻辑过期 + 互斥锁）**：

```
常规方案：设置 TTL，过期了删除，下次请求重建
你的方案：不设 TTL！在 value 里存一个"逻辑过期时间"
```

```java
// 写入时：存储"数据 + 逻辑过期时间"
RedisData redisData = new RedisData();
redisData.setData(shop);
redisData.setExpireTime(LocalDateTime.now().plusSeconds(20));
stringRedisTemplate.opsForValue().set(key, JSONUtil.toJsonStr(redisData));
// 注意：这里没设 TTL，Redis 永不过期
```

```java
// 读取时：
// 1. 拿到数据，判断逻辑过期时间
RedisData redisData = JSONUtil.toBean(json, RedisData.class);
LocalDateTime expireTime = redisData.getExpireTime();

if (expireTime.isAfter(LocalDateTime.now())) {
    // 未过期 → 直接返回
    return redisData.getData();
}

// 已过期 → 尝试获取锁（SETNX）
// 谁拿到锁，谁去重建缓存
// 没拿到锁的人，返回旧数据
```

**为什么叫"逻辑过期"**：因为数据在 Redis 里**物理上没有过期**（没设 TTL），是代码逻辑里自己去判断"这个数据该过期了"。

**流程图**：

```
请求A → 发现数据逻辑过期 → 获取锁成功 → 新线程重建 → 返回旧数据
请求B → 发现数据逻辑过期 → 获取锁失败 → 直接返回旧数据
请求C → 数据未过期 → 直接返回
```

### 5.3 缓存雪崩

**问题**：大量 key 在同一时间过期，或者 Redis 宕机，大量请求打到数据库。

**你的解决方案**：
- 没有专门处理雪崩的代码（通常是通过给 TTL 加随机值来解决）
- 但你用了**逻辑过期**方案，Redis 里的 key **永不过期**，从根本上避免了同时过期的问题

### 面试话术

> "我系统性地解决了 Redis 缓存的三大问题：穿透用空值缓存、击穿用逻辑过期+互斥锁异步重建、雪崩通过逻辑过期方案让 Redis key 永不过期来自动规避。其中缓存击穿是最复杂的方案：数据里存了一个逻辑过期时间，请求发现逻辑过期后，拿到互斥锁的线程会单独起一个线程去数据库查询并重建缓存，其他线程直接返回旧数据，这样不会影响用户体验。"

### 复习方法
- 在白板上画三张图：穿透、击穿、雪崩的请求流程图
- 自己用文字描述：如果让你解决缓存击穿，你会怎么做？（除了你的方案，还有"互斥锁方案"，就是常规 TTL+同步等锁）

## 6. 缓存双写一致性

### 问题

数据库（MySQL）和缓存（Redis）都有数据。用户更新店铺信息时，DB 改了，Redis 也要同步更新。但 DB 和 Redis 是两个独立的系统，如何保证最终一致？

### 方案选择

业界常见的方案：
1. **先更新缓存，再更新数据库** → 不行，缓存更新成功了 DB 失败，数据不一致
2. **先更新数据库，再更新缓存** → 有并发问题（写写冲突）
3. **先删除缓存，再更新数据库** → 有并发问题（读写冲突）
4. **先更新数据库，再删除缓存**（Cache Aside Pattern）← **你的方案**

### 你的代码

```java
@Transactional(rollbackFor = Exception.class)
public Result update(Shop shop) {
    // 1. 先更新数据库
    updateById(shop);
    // 2. 再删除缓存（下次读取时重建）
    stringRedisTemplate.delete(CACHE_SHOP_KEY + id);
    return Result.ok();
}
```

**为什么是"删除缓存"而不是"更新缓存"**？因为删除更简单，且避免了并发写冲突。下次读取时自然会从 DB 查询并写入缓存。

**@Transactional 的作用**：如果更新 DB 成功但删除 Redis 失败，事务会回滚，保证数据一致。

### 面试话术

> "我采用 Cache Aside Pattern，先更新数据库再删除缓存。更新数据库和删除缓存都在同一个 @Transactional 事务里，如果删除缓存失败，事务回滚，数据库也不会更新。虽然极端情况下（缓存删除成功但事务提交失败）会有短暂不一致，但概率极低，可以接受。"

## 7. 分布式ID生成器

### 为什么需要自定义ID？

表的主键一般是自增 ID。但存在两个问题：
1. **分库分表**：多张表各自自增，会生成重复 ID
2. **订单号需要规律**：像电商平台的订单号看着有规律（比如包含时间信息）

### RedisWorker 的实现原理

```java
public Long nextId(String keyPrefix) {
    // 1. 生成时间戳部分（高32位）
    long nowSeconds = LocalDateTime.now().toEpochSecond(ZoneOffset.UTC);
    long timestamp = nowSeconds - BEGIN_TIMESTAMP;   // 相对时间

    // 2. 生成序列号部分（低32位）
    String date = now.format(DateTimeFormatter.ofPattern("yyyy:MM:dd"));
    // 关键：按天分 key，每天从0开始自增
    long count = stringRedisTemplate.opsForValue()
        .increment("icr:" + keyPrefix + ":" + date);

    // 3. 拼接：时间戳左移32位 | 序列号
    return timestamp << COUNT_BITS | count;
}
```

### 结果示例

```
假设 BEGIN_TIMESTAMP = 2022-01-01 00:00:00
当前时间 = 2022-06-01 12:00:00，差约 13000000 秒
当天第 250 个订单

结果 = 13000000 << 32 | 250 = 一个很大的 long 整数
二进制: [时间戳部分(32bit)] [序列号部分(32bit)]
```

### 优点

- **全局唯一**：Redis INCR 是原子操作，分布式下不重复
- **趋势递增**：时间戳在高位，整体趋势向上
- **业务含义**：可以从 ID 中读出大致时间
- **高性能**：Redis 单机 QPS 可达数万

### 面试话术

> "我实现了一个基于 Redis 的全局 ID 生成器。ID 由 64 位组成：高 32 位是秒级时间戳偏移，低 32 位是 Redis INCR 生成的当天自增序列。这样既保证了全局唯一和趋势递增，又为将来的分库分表做了准备。按天分 key 也方便统计每天的单量。"

## 8. 分布式锁 — 从原理到实战

### 8.1 为什么需要分布式锁？

```
场景：用户秒杀下单，一人一单
如果你的服务只有一台服务器 → synchronized 就够了
如果你的服务有多台服务器 → synchronized 只锁住当前 JVM，其他服务器还能并发下单
                         → 需要一个「所有服务器都能看到的锁」
                         → 用 Redis 做这个"锁"
```

### 8.2 SimpleRedisLock（自制锁）

#### 加锁

```java
public boolean tryLock(long timeoutSec) {
    String threadId = ID_PREFIX + Thread.currentThread().getId();
    // SETNX: key 不存在才设置成功，返回 true
    Boolean success = stringRedisTemplate.opsForValue()
        .setIfAbsent("lock:" + name, threadId, timeoutSec, TimeUnit.SECONDS);
    return Boolean.TRUE.equals(success);
}
```

**注意**：加锁时传了 `threadId` 作为 value，作用见下文"解锁"。

#### 解锁 — 为什么要用 Lua 脚本

你初始版本是：

```java
// ❌ 问题代码
public void unlock() {
    String threadId = ID_PREFIX + Thread.currentThread().getId();
    String id = stringRedisTemplate.opsForValue().get("lock:" + name);
    if (threadId.equals(id)) {
        stringRedisTemplate.delete("lock:" + name);
    }
}
```

**问题**：`get` 和 `delete` 是两步操作，不是原子的。假设：
1. 线程 A 执行 `get`，拿到 id = "A"
2. 此时锁刚好到期自动释放
3. 线程 B 获取锁成功，value = "B"
4. 线程 A 继续执行 `delete`，**把线程 B 的锁删了**！

**解决**：用 Lua 脚本保证判断+删除是原子操作：

```lua
-- unlock.lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
end
return 0
```

### 8.3 SimpleRedisLock 的不足

1. **没有自动续期**：设置 10 秒过期，如果业务执行了 15 秒，锁就提前释放了
2. **非可重入**：同一个线程不能重复获取同一把锁
3. **没有等待机制**：拿不到就直接返回失败

这些不足就是引入 Redisson 的原因。

### 面试话术

> "我实现了一个基于 Redis SETNX 的分布式锁。加锁时用 SETNX 并设置过期时间防止死锁。解锁时使用 Lua 脚本保证判断锁归属和删除锁的原子性，防止误删其他线程的锁。但这个锁缺少自动续期和可重入能力，所以后续换成了 Redisson。"

### 复习方法
- 在白板上画出：线程A获取锁 → 执行业务 → 线程B尝试获取锁 → 线程A释放锁的全过程
- 想清楚"为什么不用 synchronized"→"为什么 get+delete 有问题"→"为什么 Lua 能解决"

## 9. 秒杀系统 — 高并发全链路设计

这是你项目的**核心模块**，也是面试最能讲技术深度的部分。

### 9.1 秒杀的挑战

```
1. 库存扣减不能超卖（100个商品不能卖101单）
2. 一人一单限制（同一用户只能买一次）
3. 高并发（瞬间大量请求）
4. 一致性（不能丢单、不能重复）
```

### 9.2 你的方案演进（三个版本）

#### V1：纯数据库方案（注释掉的代码）

```java
// 1. 查秒杀券，判断时间、库存
// 2. synchronized(用户ID) 锁住一人一单
// 3. @Transactional + update ... where stock > 0 扣库存
// 4. 创建订单
```

**问题**：每次请求都打数据库，并发高时数据库扛不住。

#### V2：Lua + 阻塞队列异步下单（注释掉的代码部分）

```java
// 1. Lua 脚本判断库存+一人一单（Redis 内存操作）
// 2. 满足条件 → 把订单信息丢进 JVM 阻塞队列
// 3. 后台线程从队列取数据，写入 DB
```

**问题**：阻塞队列在 JVM 内存里，服务重启或宕机会**丢数据**。

#### V3（最终版）：Lua + Redis Stream 异步下单

```
用户请求 → Lua脚本(Redis内存操作)
             ↓ 成功 → Redis Stream 写入消息
                        ↓ 后台线程消费 Stream 消息
                           ↓ 写入 MySQL（真正的持久化）
```

### 面试话术（用演进的方式讲）

> "秒杀模块我做了三个版本的迭代。第一版直接操作数据库，压测发现 QPS 太低，数据库连接打满。第二版把库存扣减放到 Redis 中用 Lua 脚本，然后订单信息塞到 JVM 阻塞队列异步落库，但服务重启会丢消息。最终版把阻塞队列换成了 Redis Stream，利用 Redis 的持久化机制保证消息可靠，且 Stream 支持消费者组，方便后续扩展。最终实现从用户请求到数据库落库的完全异步化。"

## 10. Lua 脚本 — Redis 原子操作

### 为什么用 Lua？

**场景**：秒杀时要完成"检查库存 → 检查是否已买 → 扣库存 → 记录用户" 这四步操作。如果分四次请求 Redis，中间可能被其他请求插入。

**Lua 脚本的特性**：
- Redis 执行 Lua 脚本是**原子操作**（整个脚本执行时不会插入其他命令）
- 减少网络开销（一次请求代替多次）
- 逻辑在服务端执行，安全

### seckill.lua 详解

```lua
-- ARGV 是从 Java 传过来的参数
local voucherId = ARGV[1]    -- 优惠券ID
local userId = ARGV[2]       -- 用户ID
local orderId = ARGV[3]      -- 订单ID（在Java中已用分布式ID生成器生成）

-- Redis key 的拼接
local stockKey = 'seckill:stock:' .. voucherId     -- 库存 key
local orderKey = 'seckill:order:' .. voucherId     -- 已下单用户集合 key

-- 1. 检查库存
local stock = tonumber(redis.call('get', stockKey))
if (not stock or stock <= 0) then
    return 1    -- 库存不足
end

-- 2. 检查是否已买（一人一单）
if (redis.call('sismember', orderKey, userId) == 1) then
    return 2    -- 重复下单
end

-- 3. 扣库存（用 set 记录已买的用户，就是一人一单的保证）
redis.call('incrby', stockKey, -1)        -- 库存 -1
redis.call('sadd', orderKey, userId)      -- 加入已购买集合

-- 4. 发送消息到 Redis Stream（异步落库）
redis.call('xadd', 'stream.orders', '*',
    'userId', userId, 'voucherId', voucherId, 'id', orderId)

return 0  -- 成功
```

### Java 端调用

```java
Long result = stringRedisTemplate.execute(
    SECKILL_SCRIPT,           // Lua 脚本
    Collections.emptyList(),  // KEYS 数组（没有用 KEYS，所以传空）
    vocherId.toString(),       // ARGV[1]
    userId.toString(),         // ARGV[2]
    String.valueOf(orderId)    // ARGV[3]
);
int r = result.intValue();
if (r == 0) {
    // 秒杀成功！
    return Result.ok(orderId);
} else if (r == 1) {
    return Result.fail("库存不足");
} else {
    return Result.fail("不能重复下单");
}
```

### 面试话术

> "我使用 Lua 脚本将库存扣减、一人一单判断、消息发送封装成一个原子操作。相比多次 Redis 操作，Lua 脚本减少了网络往返，且保证了原子性，不会出现并发超卖。脚本执行完成后返回 0/1/2 三种状态，Java 端根据返回值判断秒杀结果。"

## 11. Redis Stream — 消息队列与异步削峰

### 为什么需要消息队列？

```
秒杀场景：
用户点击 → Lua 脚本 → 扣库存成功 → 然后呢？要不要立即写数据库？

如果立即写数据库 → 高并发时数据库压力大，锁冲突多
如果不立即写 → 怎么保证最终会写？

答案：把写操作变成异步任务，先快速响应"秒杀成功"，后台慢慢落库
```

### Redis Stream VS JVM 阻塞队列

| 对比项 | JVM 阻塞队列 | Redis Stream |
|--------|-------------|--------------|
| 持久化 | 无，进程重启丢失 | 有，Redis RDB/AOF 持久化 |
| 容量 | JVM 内存限制 | Redis 内存 |
| 消费确认 | 无 | 支持 ACK 确认 |
| 重试机制 | 无 | 支持 Pending List |

### 你的实现

#### 1. 生产者（Lua 脚本中）

```lua
redis.call('xadd', 'stream.orders', '*',
    'userId', userId, 'voucherId', voucherId, 'id', orderId)
```

`XADD stream.orders *`：向 `stream.orders` 这个 Stream 追加一条消息，`*` 表示让 Redis 自动生成消息 ID。

#### 2. 消费者（Java 后台线程）

```java
@PostConstruct  // 项目启动时自动运行
private void init() {
    SECKTLL_ORDER_EXECUTOR.submit(new VoucherOrderHandler());
}

private class VoucherOrderHandler implements Runnable {
    public void run() {
        while (true) {
            // 1. 读取 Stream 新消息（阻塞等待，最多 2 秒）
            List<MapRecord<String, Object, Object>> list =
                stringRedisTemplate.opsForStream().read(
                    Consumer.from("g1", "c1"),    // 消费者组 g1, 消费者 c1
                    StreamReadOptions.empty().count(1).block(Duration.ofSeconds(2)),
                    StreamOffset.create("stream.orders", ReadOffset.lastConsumed())
                );

            // 2. 处理订单
            handleVoucherOrder(voucherOrder);

            // 3. ACK 确认（告诉 Redis 这条消息已处理完）
            stringRedisTemplate.opsForStream()
                .acknowledge("stream.orders", "g1", mapRecord.getId());
        }
    }
}
```

#### 3. 容错：Pending List 机制

```java
private void handlePendingList() {
    // 读取 Pending List（已投递但未 ACK 的消息）
    List<MapRecord<String, Object, Object>> list =
        stringRedisTemplate.opsForStream().read(
            Consumer.from("g1", "c1"),
            StreamReadOptions.empty().count(1),
            StreamOffset.create("stream.orders", ReadOffset.from("0"))
            // ReadOffset.from("0") = 只读 Pending List 的消息
        );
    // ... 处理 + ACK
}
```

**什么时候触发**：正常消费抛异常时，会在 catch 块中调用 `handlePendingList()`，把之前没 ACK 的消息重新处理一遍。

### 面试话术

> "秒杀的核心瓶颈在数据库写入。我用 Redis Stream 作为消息队列，Lua 脚本扣库存后通过 XADD 发送一条消息到 Stream，后台线程从 Stream 消费消息异步写入数据库。Stream 支持消费者组和 ACK 机制，即使消费过程中发生异常，也可以通过 Pending List 重新消费，不会丢失订单。相比 JVM 阻塞队列，Redis Stream 具备持久化能力，服务重启不丢消息。"

### 复习方法
- 画时序图：用户 → Controller → Lua（Redis）→ Stream → 后台线程 → MySQL
- 理解"削峰"的含义：把瞬间的并发写请求"削平"成稳定的异步写入

## 12. Redisson — 生产级分布式锁

### Redisson 解决了什么？

你的 SimpleRedisLock 缺点：
1. **没有自动续期**：锁 10 秒过期，业务执行 15 秒 → 锁提前释放
2. **非可重入**：同一线程不能重入

### Redisson 的特点

```java
// 获取 Redisson 客户端
@Bean
public RedissonClient redissonClient() {
    Config config = new Config();
    config.useSingleServer()
        .setAddress("redis://127.0.0.1:6379")
        .setPassword("123456");
    return Redisson.create(config);
}

// 使用
RLock lock = redissonClient.getLock("lock:order:" + userId);
boolean islock = lock.tryLock();  // 获取锁
try {
    proxy.createVoucherOrder(voucherOrder);
} finally {
    lock.unlock();
}
```

### Redisson 锁的核心机制 — WatchDog（看门狗）

```
1. 加锁成功 → Redis 写入一个 key，默认 30 秒过期
2. Redisson 启动一个后台线程（WatchDog）
3. 每隔 10 秒检查一次：锁还在不在？还在就续期到 30 秒
4. 业务执行完成 → 手动释放锁
5. 如果宕机 → WatchDog 也停了 → 锁最多 30 秒自动释放
```

**好处**：你不用担心业务执行时间超过锁过期时间，WatchDog 帮你自动续期。

### 可重入性

```java
lock.tryLock();  // 第一次获取锁成功
lock.tryLock();  // 同一个线程再次获取 → 计数+1，成功
lock.unlock();   // 计数-1
lock.unlock();   // 计数为0，真正释放
```

### 使用场景在你的代码中

```java
// VoucherOrderServiceImpl.java - handleVoucherOrder
private void handleVoucherOrder(VoucherOrder voucherOrder) {
    Long userId = voucherOrder.getUserId();
    // 使用 Redisson 锁住同一个用户的一人一单
    RLock lock = redissonClient.getLock("lock:order:" + userId);
    boolean islock = lock.tryLock();
    if (!islock) {
        log.error("不允许重复下单");
        return;
    }
    try {
        proxy.createVoucherOrder(voucherOrder);  // 通过代理调用事务方法
    } finally {
        lock.unlock();
    }
}
```

**为什么锁的是 `userId` 不是其他东西**？因为"一人一单"的粒度是**用户级别**，同一个用户不能并发下单，不同用户可以同时下单。

### 面试话术

> "我使用 Redisson 作为生产级的分布式锁方案。相比自制的 SETNX 锁，Redisson 提供了自动续期的 WatchDog 机制和可重入能力。在秒杀场景中，我用 Redisson 以用户 ID 为粒度加锁，保证了同一用户不会并发创建多个订单，同时不同用户之间互不影响。"

## 14. 全局异常处理

### 为什么需要全局异常处理？

如果没有统一处理，每个 Controller 都要 try-catch，代码重复。而且直接抛给 Spring Boot 默认的错误页面，返回的是 HTML 而不是 JSON。

### 你的实现

```java
@Slf4j
@RestControllerAdvice  // 控制器增强：拦截所有 Controller 抛出的异常
public class WebExceptionAdvice {

    @ExceptionHandler(RuntimeException.class)  // 拦截所有 RuntimeException
    public Result handleRuntimeException(RuntimeException e) {
        log.error(e.toString(), e);  // 记日志
        return Result.fail("服务器异常");  // 返回统一 JSON
    }
}
```

**设计点**：
- `@RestControllerAdvice` = `@ControllerAdvice` + `@ResponseBody`，直接把结果转 JSON
- 只拦截 `RuntimeException`，受检异常（如 `IOException`）不会被处理
- 日志记录完整的异常堆栈（`log.error` 传 e 而不是 `e.getMessage()`）
