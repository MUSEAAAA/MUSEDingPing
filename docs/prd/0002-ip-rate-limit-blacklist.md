# PRD-002：IP 限流与黑名单防刷模块

## Problem Statement

当前秒杀系统依赖"一人一单"规则防刷——同一用户 ID 对同一秒杀券只能下一单。但黄牛可以批量注册数百个账号，从同一台服务器高频调用秒杀接口。用户维度的约束对这种攻击无效。系统缺少 IP 维度的流量控制，秒杀接口在高频脚本攻击下没有任何防护。

## Solution

在秒杀请求进入业务逻辑之前，新增一个 IP 维度的拦截层。每个 IP 每秒最多 10 次请求，累计被限流 30 次后自动进入黑名单封禁 1 小时。被拦截的请求直接返回 HTTP 429，不消耗业务资源（token 解析、Lua 执行、Stream 消息）。

## User Stories

1. 作为一个正常用户，我以正常速度刷新页面抢券，请求不会被 IP 限流误拦截
2. 作为一个正常用户，我被限流后下一秒计数器重置，请求自动恢复
3. 作为一个正常用户，我偶尔抢快了被限流几次，不会进黑名单
4. 作为一个使用脚本刷券的黄牛，我的 IP 每秒第 11 次请求会被拒绝并收到 HTTP 429
5. 作为一个被多次限流的脚本 IP，累计 30 次后自动进入黑名单，后续请求直接拒绝
6. 作为一个被封禁的 IP 用户，我能从错误信息中看到剩余封禁时间
7. 作为一个被封禁的 IP 用户，封禁时间结束后我可以正常请求秒杀接口
8. 作为一个运维人员，限流阈值和黑名单阈值可以通过配置修改，无需重启应用
9. 作为一个运维人员，黑名单中的 IP 在 TTL 到期后自动解封，无需手动操作
10. 作为一个访问其他接口的用户，浏览店铺、查看笔记等非秒杀请求不受 IP 限流影响
11. 作为一个被限流的用户，我收到的错误信息和黑名单用户不同，能区分当前状态
12. 作为一个系统的其他模块（优惠券管理、店铺查询），秒杀以外的所有接口性能不受防刷模块影响

## Implementation Decisions

### 新增模块：RateLimitInterceptor

- 新增一个 Spring Interceptor，`order=0`，在最外层拦截
- 拦截路径只匹配秒杀下单接口，不拦截其他路径
- 现有 `RefreshTokenInterceptor` 和 `LoginInterceptor` 的 order 分别后移到 1 和 2

### Lua 脚本：单次 Redis 往返完成限流 + 计数 + 黑名单

限流检查、累计计数器、黑名单写入三个操作用一个 Lua 脚本在 Redis 中原子执行，避免多次网络 IO 带来的竞态窗口。脚本返回值编码四种结果：通过、限流、黑名单命中、限流且触发黑名单。

### 固定窗口算法

使用固定窗口（key 包含秒级时间戳），而非滑动窗口。固定窗口的边界突刺在 10 次/秒阈值下影响有限，且实现更简单，面试时可主动解释 trade-off。

### 黑名单存储

黑名单存 Redis，不写 MySQL。理由是封禁有时效（1 小时），服务重启丢失黑名单可接受。风控审计日志写入应用日志即可。

### Redis Key 设计

三级 key：秒级计数器（TTL 2s）、累计被限流次数（TTL 同黑名单 TTL）、黑名单标记（TTL 1 小时）。

### 可配置项

阈值通过 `application.yaml` 注入，支持运行时修改配置中心（如 Nacos/Apollo）动态调整。

### 错误响应

限流和黑名单返回不同的错误信息。黑名单错误信息包含剩余封禁分钟数，通过 Redis `TTL` 命令获取。

### 配置项默认值

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 每秒限流阈值 | 10 | 同一 IP 每秒最大请求数 |
| 黑名单触发阈值 | 30 | 累计被限流多少次后拉黑 |
| 黑名单时长 | 3600s | 封禁持续秒数 |

## Testing Decisions

### 测试原则

只测外部行为，不测实现细节。Lua 脚本测返回值契约，Interceptor 测 HTTP 响应码和 errorMsg，集成测试测并发下的计数准确性。

### 测试缝

整个防刷模块只接触一个缝：新增的 `RateLimitInterceptor`。不修改现有 Controller、Service、Lua 脚本。测试用 Mock Redis（内嵌 Redis 或 mock `StringRedisTemplate`）注入 Interceptor，验证四种 Lua 返回值对应正确的 HTTP 响应。

### 测试层级

| 层级 | 测什么 | 工具 |
|------|--------|------|
| Lua 脚本 | 四种返回值的触发条件 | 内嵌 Redis 或单元测试 mock |
| Interceptor | 拦截/放行分支、错误信息格式 | MockMvc + mock Redis |
| 并发集成测试 | 并发下计数器准确、黑名单触发 | CountDownLatch + 真实 Redis |

### 现有测试参考

项目已有 `HmDianPingApplicationTests` 使用 `CountDownLatch` + `ExecutorService` 做并发测试，防刷模块的并发测试沿用此模式。
